# patchwork

코드 작업 시 따를 규율. 두 축: **맥락 보존** + **독립 검증**.

---

## 0. 작성 전 명료화

코드를 쓰기 전에 다음을 만족시킨다:

1. **가정 명시** — 불확실하면 묻는다. 묵묵히 하나를 고르지 않는다.
2. **다중 해석 표면화** — 요청이 여러 방향으로 읽히면 모두 제시한다.
3. **더 단순한 대안 제시** — 있으면 말한다. 근거 있으면 밀어낸다.
4. **정지·질문** — 불분명하면 멈추고 무엇이 불분명한지 이름을 붙인 뒤 묻는다.

판정 한 줄:
> *"이 해석이 틀렸다면 사용자가 다시 작업을 시키게 되는가?" → ✓ 이면 확인 후 시작.*

## 1. 모듈 블랙박스

### CONTEXT.md 의 존재 = 블랙박스 선언

- `CONTEXT.md` 가 있는 디렉토리는 **블랙박스로 취급**. 내부 파일을 외부에서 직접 import 하지 않는다. 진입점만 통과.
- `CONTEXT.md` 가 없는 디렉토리는 일반 폴더. 자유롭게 다룬다.

### CONTEXT.md 의 고정 구조

```markdown
# <모듈명>

## 진입점
- `<경로>` — <무엇을 노출하는지>

## 테스트
- 유닛: `<경로>`
- E2E: `<경로>`

## 용어
- **<용어>** — <정의>
```

- 세 섹션 고정. 항목이 없어도 섹션은 유지하고 "없음" 으로 표기한다.
- 진입점 파일명은 언어·프레임워크 관례에 맡긴다 (`index.ts`, `__init__.py`, `lib.rs` 등). CONTEXT.md 가 경로만 가리킨다.
- 용어 사전은 **bullet list** 형식 고정: `- **용어** — 정의`.

### 분할 트리거

**CONTEXT.md 가 300줄을 넘으면 모듈을 분할한다.** 이 규칙은 CONTEXT.md 자체의 크기만 본다 (모듈 내부 코드 줄 수는 대상 아님). CONTEXT.md 하나로 모듈 전체를 파악할 수 있어야 한다는 원칙이 깨지는 시점이 분할 시점.

### 용어 충돌 감지

- 같은 레포의 다른 `CONTEXT.md` 에서 **같은 단어가 다른 정의**로 등장하면 위반. 조사는 CONTEXT.md 간 용어 대조로 한다.
- 한 모듈 내부에서는 **하나의 개념에 하나의 이름**만 쓴다. 별명 금지.

## 2. 의도 주석 (`// WHY:`)

### 형식

```ts
// WHY: 과거 X 방식에서 Y 버그 발생, 그래서 Z 로 변경
```

- 태그는 `// WHY:` 고정 (언어별 주석 기호로 시작: `#`, `/* */` 등).
- 다중 라인은 첫 줄에만 태그를 쓰고 다음 줄부터 들여써서 이어간다.

```ts
// WHY: 단순 map 쓰면 병렬성으로 race condition.
//      Promise.all 로 묶어야 순서 보장.
```

### 사용 조건

다음 중 하나라도 해당하면 `// WHY:` 를 붙인다:

1. **단순해 보이는 방식을 일부러 택하지 않은** 자리
2. **직관과 반대되는 결정**
3. **과거 버그·이슈 대응 workaround**
4. **성능·보안·호환성 제약**에서 비롯된 선택

판정 한 줄:
> *"이 주석이 없으면, 이 코드를 처음 보는 에이전트가 '더 단순하게 고칠 수 있겠다' 하고 롤백할 가능성이 있는가?"*

### WHY 라인 수정 규칙

`WHY:` 가 붙은 라인을 수정·삭제할 때는 **주석의 조건이 아직 유효한지 먼저 검증**한다. 유효하지 않게 되었다면 주석과 코드를 함께 갱신한다. 주석만 남기고 코드를 바꾸는 일 금지.

### 면제 선언 장치

특정 규칙의 면제를 주장하려면 **파일 상단 또는 해당 라인에 `WHY(<면제 항목>):`** 형식으로 사유를 명시한다. 선언 없이 면제 주장 불가.

```ts
// WHY(테스트 면제): 1회성 마이그레이션, 실행 후 삭제
// WHY(평가 면제): 주석 오타 수정만
```

### 변경 범위 외 불개입

WHY 주석이 없는 라인이라도 요청 범위 밖이면 건드리지 않는다:

- 인접 코드·주석·포맷을 "개선" 하지 않는다.
- 깨지지 않은 것을 리팩터하지 않는다.
- 기존 스타일을 유지한다 (본인 취향으로 바꾸지 않는다).
- 무관한 죽은 코드는 **언급만**, 삭제하지 않는다.
- 본인 변경이 만든 orphan(미사용 import·변수·함수) 은 제거한다. 기존 dead code 는 요청 없으면 유지한다.

판정 한 줄:
> *"변경된 모든 라인은 사용자 요청으로 직접 추적되는가?"*

## 3. 병렬 작업

### 작업 병렬은 3 조건 모두 충족 시에만 허용

1. **입력 독립** — 각 하위 작업이 다른 하위 작업의 산출물을 입력으로 요구하지 않는다.
2. **출력 독립** — 각 결과가 다른 결과와 결합되어야만 의미가 생기지 않는다.
3. **맥락 재파악 없음** — 동일한 모듈·코드베이스 설명을 각 에이전트에게 반복 주입하지 않아도 된다.

**하나라도 ✗ → 순차 실행 또는 단일 에이전트.**

### 판정 예시

| 작업 | 1 | 2 | 3 | 판정 |
|---|---|---|---|---|
| 독립 3 프로젝트 스캐폴딩 | ✓ | ✓ | ✓ | 허용 |
| 20 파일에 동일한 기계적 포맷 변환 | ✓ | ✓ | ✓ | 허용 |
| 한 기능을 모듈별로 3 서브에이전트 분담 | ✗ | ✗ | ✗ | 금지 |
| 파일 간 일관성 필요한 리팩터 | ✗ | ✗ | ✗ | 금지 |
| 관점 병렬 (동일 산출물 교차검증) | ✓ | ✓ | — | 허용 (특례) |

### 관점 병렬 특례

같은 산출물을 서로 다른 에이전트·모델이 독립적으로 검토하는 구조. **맥락 중복이 의도된 설계**이므로 조건 3 은 적용하지 않는다. 프로토콜은 §5 참조.

### 병렬 선언 (권장)

작업 병렬을 택하면 메인 에이전트가 세 조건을 어떻게 충족하는지 선언한다. 사후 감사용.

```
# 작업 병렬 선언
- 입력 독립: <근거>
- 출력 독립: <근거>
- 맥락 독립: <근거>
```

## 4. TDD 판정

### 두 필터

다음 **두 조건 모두 충족** 시 테스트 우선 작성 의무:

1. **지속성** — 이 코드가 지속적으로 사용되는가? (실행·참조·로드 포함. 한 번 쓰고 버리지 않는가)
2. **검증 가능성** — 입력이 있고, 출력 또는 관찰 가능한 상태 변화가 있으며, 둘의 관계를 코드로 단언할 수 있는가? (관찰 가능한 상태 변화 = 반환값, DB 쓰기, 외부 호출, 로그 등. 순수 타입·상수 선언은 해당 없음)

둘 다 ✓ → TDD 의무. 하나라도 ✗ → 면제.

### 휴리스틱 (판정 애매할 때만 참조)

| 코드 | 지속성 | 검증 가능성 | TDD |
|---|---|---|---|
| 비즈니스 로직·알고리즘 | ✓ | ✓ | 의무 |
| 유틸·헬퍼 함수 | ✓ | ✓ | 의무 |
| DB·HTTP 통합 코드 | ✓ | ✓ | 의무 (통합 테스트) |
| UI 컴포넌트 (로직 있음) | ✓ | ✓ | 의무 |
| 순수 타입·상수 선언 | ✓ | ✗ | 면제 |
| 런타임 검증 붙은 설정 (zod 스키마, env 파서) | ✓ | ✓ | 의무 |
| 일회성 마이그레이션 스크립트 | ✗ | — | 면제 (선언 필요) |
| 탐사·프로토타입 스크립트 | ✗ | — | 면제 (선언 필요) |

**휴리스틱과 두 필터가 충돌하면 두 필터가 상위.**

### 면제 선언

일회성·프로토타입으로 면제받으려면 **파일 주석에 `WHY(테스트 면제):` 명시**. 선언 없이 면제 불가. 유지하기로 결정되는 순간 재판정.

### 테스트 레벨 선택

모듈 블랙박스 경계에 맞춘다:
- **모듈 내부** → 단위 테스트
- **모듈 간 경계** → 통합 테스트
- **시스템 진입점** → E2E 테스트

CONTEXT.md `## 테스트` 섹션에 각 레벨의 경로를 기록한다.

### 요청 → 검증 가능 목표 변환

§4 TDD 대상으로 판정된 요청을 테스트 형태로 변환한 뒤 §5 평가자에게 수용 기준으로 전달한다:

| 원 요청 | 변환 |
|---|---|
| "검증 추가" | "잘못된 입력용 테스트 작성 후 통과시키기" |
| "버그 수정" | "재현 테스트 작성 후 통과시키기" |
| "X 리팩터" | "전·후 테스트 통과 보장" |

약한 기준("동작하게") 은 §5 평가자가 판정 불가. 강한 기준만 전달한다.

변환은 §0 (작성 전 명료화) 가 완료된 후에 수행한다. 요청의 의미 자체가 불분명하면 변환을 먼저 시도하지 말고 §0 으로 돌아가 묻는다.

## 5. 평가자 호출

### 호출 시점

다음 중 하나라도 해당하면 평가자 호출 의무:

1. **코드 로직이 변경된 경우** (단순 주석·포맷·오타 수정 제외)
2. **TDD 대상 코드가 생성·변경된 경우** (§4 의무 대상)
3. **기존 `WHY:` 주석이 있는 라인이 수정된 경우** (롤백 감시)

면제: `WHY(평가 면제):` 주석으로 사유 선언 시.

### 평가자에게 전달할 정보

**전달 (See)** — 객관적 사실:
- 사용자 요청 원문
- 수용 기준 (항목화된 성공 조건)
- 영향 모듈의 `CONTEXT.md`
- 변경 diff (또는 영향 파일 전체)
- 테스트 실행 결과
- 코드 내 `WHY:` 주석 (코드 안에 박힌 객관 문서)

**차단 (Hide)** — 작성자의 서사:
- 작성자의 구두/문서 설명·정당화 (코드 밖)
- 작성자가 검토·기각한 대안
- 과거 시도·실패 로그

핵심 원칙:
> *정보가 "코드와 함께 버전 관리되는가" 로 분리. 코드에 박혀 있으면 객관, 코드 밖에서 말로 하면 서사.*

### 관점 병렬 프로토콜

**서로 다른 2 평가자**를 호출한다:

- 평가자 A: Claude subagent (다른 역할 agent — 예: `critic`, `architect`)
- 평가자 B: 다른 모델 계열 (Codex 등)

**동일 모델·동일 agent 복제 금지** (같은 실패 공유).

### 결과 결합

- 둘 다 Pass → 통과
- 둘 다 Fail → 재작성
- 의견 불일치 → **3번째 평가자 호출 또는 사용자 결정**

## 용어 사전

### 모듈 관련
- **모듈** — 블랙박스 취급의 단위. `CONTEXT.md` 가 있는 디렉토리.
- **진입점** — 모듈의 외부 경계 파일. CONTEXT.md `## 진입점` 섹션에 경로가 기록됨.
- **블랙박스** — 특정 단위부터 그 내부를 열지 않고 외부 계약으로만 다루는 방식.

### 주석·문서
- **의도 주석 / `WHY:` 주석** — "왜 이렇게 했는가" 를 기록한 주석.
- **롤백** — 이전에 변경한 코드를 과거의 문제 있던 상태로 되돌리는 행위.
- **면제 선언** — `WHY(<항목>):` 형식으로 규칙 면제를 주석에 명시하는 것.

### 검증
- **작성자** — 코드 작업을 수행하는 주체.
- **평가자** — 작성자와 별도로 결과물을 평가하는 주체. subagent 또는 다른 모델.
- **치팅** — 에이전트가 평가를 우회하는 행위 (예: 작성자 설명이 평가자에게 흘러들어가 정당화에 이용되는 것).

### 병렬
- **작업 병렬** — 서로 다른 작업을 여러 서브에이전트가 나누어 수행. 3 조건 충족 시에만 허용.
- **관점 병렬** — 같은 산출물을 여러 에이전트·모델이 독립적으로 검토. 허용.

이 스킬의 관찰·원리·대안 검토·기각된 접근은 같은 폴더의 `NOTES.md` 에 기록되어 있다. 규칙에 의문이 생기면 먼저 `NOTES.md` 를 참조한다.
