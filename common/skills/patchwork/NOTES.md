# patchwork

코드 작업 시 따를 개발 규율. 누더기처럼 덧대는 가드레일 — 강제 프레임워크가 아니라 경계만 표시.

## 두 축의 근거

**맥락은 새어나간다**: 에이전트는 컨텍스트가 없으면 재생성한다. 없는 함수를 재작성하고, 이유를 모르는 결정을 되돌리고, 서브에이전트끼리 같은 용어를 다르게 쓴다.

**자기 자신은 못 믿는다**: 작성자는 자기 실수를 못 본다. 단일 모델은 같은 방식으로 틀리고, 에이전트는 평가를 치팅으로 우회할 수 있다.

---

## 맥락 보존

### 모듈 블랙박스

**관찰**: 함수를 재사용하라고 아무리 잘 지시해도, 에이전트는 그게 존재하는지 모르면 재작성한다.

**원칙**:
- 특정 단위부터 그 내부를 **블랙박스**처럼 여긴다
- 그 단위로 **유닛 및 E2E 테스트**가 명확히 되어야 한다
- 진입점에는 **상세한 주석과 파일 링크**가 담겨 있어야 한다
- 모듈 폴더에 **별도의 md 파일**을 두어 에이전트가 모두 파악할 수 있을 만큼 그 크기가 충분히 작아야 한다

**모듈의 단위**: 라이브러리 자체일 수도 있고, 소스코드 몇 개가 담긴 디렉토리일 수도 있다.

**예시**:
- Good — 진입점 파일에 모듈이 제공하는 함수 목록과 내부 파일 링크가 정리되어 있어, 에이전트가 내부에 들어가지 않고도 무엇이 존재하는지 파악 가능.
- Bad — 외부 코드가 모듈 내부 함수를 직접 import·수정. 진입점이 모호해 에이전트가 "이 기능 없네" 판단하고 재작성.

**구체화 예정**: md 파일의 내부 구조 / "파악 가능한 크기"의 기준 / 분할 트리거.

### 바운디드 컨텍스트 — DDD에서 차용

DDD 전체의 범용성은 미검증 (아래 **미검증 영역**). 두 용어만 차용한다.

**Bounded Context** — 하나의 도메인 모델이 일관되게 적용되는 경계를 가리키는 DDD 개념. 같은 단어가 경계 바깥에서는 다른 의미일 수 있다.

차용한 규칙: 모듈마다 고유의 용어 사전을 둔다. 각 모듈의 md 파일에 그 모듈에서의 의미를 고정한다.

**Ubiquitous Language** — 팀·코드·문서가 같은 어휘를 일관되게 쓴다는 DDD 원칙.

차용한 규칙: 모듈 내에서는 하나의 개념에 하나의 이름만 사용한다. 별명을 쓰지 않는다.

**구체화 예정**: 용어 사전의 형식 / 규칙 위반 감지 방식 / 평가 체크리스트와의 연결.

### 의도 주석

**원칙**: 왜 이렇게 하였는가란 **의도 자체가 주석에** 담겨 있어야 한다.

**이유**: 맥락 잃은 에이전트는 이전에 변경한 코드를 **과거의 문제 있던 상태로 되돌리려** 한다.

**예시**:
- Good — "과거 X 방식 사용 시 Y 버그 발생, 그래서 Z로 변경" 처럼 **과거의 문제와 선택 근거**를 남긴 주석. 에이전트가 롤백 시도 전에 이유를 확인 가능.
- Bad — "Z로 변경" 처럼 **무엇**만 있고 **왜**가 없는 주석. 미래의 에이전트가 동일한 과거 문제를 다시 일으키는 방향으로 되돌릴 수 있음.

**구체화 예정**: 주석의 형식과 밀도 / 의도 주석이 있는 줄의 수정 절차 / 롤백 감시를 평가자와 연결하는 방식.

### 병렬 작업

**관찰**: 단순 병렬 작업(team, swarm 등)은 최악. 완전히 병렬 작업이 아닌 경우, 각 서브에이전트는 맥락 파악이 어려워져 다시 맥락을 파악하기 위해 수많은 토큰을 낭비한다. 비슷하거나 약간 나은 결과를 위해 매우 큰 비용을 소비.

**예외 — 관점 병렬**: claude-codex는 병렬이지만 **같은 산출물을 여러 모델이 교차검증**하는 구조. 단일 모델이 하나의 잘못된 것으로 수렴하는 것을 방지하고, 평가자를 따로 두어 치팅도 방지한다.

- 작업 병렬 ✗
- 관점 병렬 ✓

**예시**:
- Bad (작업 병렬) — 하나의 기능을 3개 서브에이전트가 모듈별로 나눠 구현. 각자 전체 맥락이 없어 용어·인터페이스가 엇갈리고, 재파악 비용이 본 작업을 초과.
- Good (관점 병렬) — 동일한 산출물을 서로 다른 모델(Claude + Codex)이 각각 검토. 단일 모델이 하나의 틀린 답으로 수렴하는 것을 방지하고, 평가자 분리로 치팅도 차단.

**구체화 예정**: 병렬이 허용되는 조건의 판정 기준 / 관점 병렬의 세부 프로토콜.

## 독립 검증

### TDD 적용 범위

**원칙**: 모든 작업은 **가능한 테스트 우선 작성**이 필요.

**미정**: "메인 코드 작성 시에만" — 적용 경계 구체화 필요.
- 어디까지 "메인 코드" 인가
- 일회성 / 설정 / 프로토타입 등의 처리
- 판정 기준의 정확한 형태

### 평가자로서의 claude-codex

**역할**: claude-codex는 이 맥락에서 **평가 및 검증의 지표**로만 사용한다.

**효과**:
- 한 모델이 하나의 잘못된 것으로 수렴하는 것을 단일 모델에 비해 어느 정도 방지
- 평가자를 따로 두어 기존 작업의 **치팅** 방지

**원칙**: 모든 코드 작업 후에는 별도의 평가자(subagent 혹은 codex)에게 평가를 요청한다.

**구체화 예정**: 호출 프로토콜 / 평가자에게 전달할 정보의 범위 / 결과 처리 방침.

### 평가자에게 전달할 정보 범위

**상태**: 슬롯만 두고 내용은 미정. 평가자 구성의 한 항목으로 구체화 필요.

**열린 질문**:
- 평가자가 **무엇을 봐야 하는가** (결과물, 수용 기준, 테스트, 모듈 경계 등)
- 평가자가 **무엇을 보지 않아야 하는가** (작성 동기, 검토한 대안, 사용자 요청 맥락 등)
- 롤백 감시 같은 예외 처리

## 미검증 영역

### DDD — 나머지 요소

**관찰**: DDD는 커뮤니티에서 종종 추천되는 방법이지만 아직 실제로 검증해보지 못함. 범용성이 있는지는 생각해볼 필요.

**현재 차용 후보**: Bounded Context, Ubiquitous Language (위 **바운디드 컨텍스트** 섹션).

**나머지**: 다른 DDD 요소들의 차용 여부는 미정. 실전 경험 누적 후 판단.

## 용어 사전

### 스킬의 성격
- **patchwork / 누더기 / 조각보** — 이 스킬의 이름이자 성격. 완결된 체계가 아니라 필요할 때마다 조각을 덧대며 자라는 구조.
- **가드레일** — 강제 없이 경계만 표시하는 장치. 프레임워크의 반대 지향.
- **두 축** — patchwork 원칙의 두 계열. 맥락 보존과 독립 검증.
- **맥락 보존** — "맥락은 새어나간다" 라는 관찰에 대응하는 원칙 계열.
- **독립 검증** — "자기 자신은 못 믿는다" 라는 관찰에 대응하는 원칙 계열.
- **구체화 예정** — 각 섹션의 열린 질문. 사용자 결정 전까지 선제 구체화 금지.

### 모듈
- **모듈** — 블랙박스 취급의 단위. 라이브러리 자체일 수도, 소스코드 몇 개가 담긴 디렉토리일 수도 있다.
- **블랙박스** — 특정 단위부터 그 내부를 열지 않고 외부 계약으로만 다루는 방식.
- **진입점** — 모듈의 외부 경계가 되는 파일. 상세한 주석과 파일 링크를 담는다.

### 용어와 주석
- **바운디드 컨텍스트** — 모듈마다 고유의 용어 체계를 갖는다는 규칙. DDD에서 차용.
- **Ubiquitous Language** — 모듈 내에서 하나의 개념에 하나의 이름만 사용한다는 규칙. DDD에서 차용.
- **의도 주석** — "왜 이렇게 하였는가"를 담은 주석.
- **롤백** — 이전에 변경한 코드를 과거의 문제 있던 상태로 되돌리는 행위.
- **롤백 감시** — 의도 주석과 평가자를 연결하는 예외 처리 구조 (구체화 예정).

### 검증
- **작성자** — 코드 작업을 수행하는 주체.
- **평가자** — 작성자와 별도로 결과물을 평가하는 주체. subagent 또는 codex.
- **claude-codex** — Claude와 Codex의 쌍 구조. 이 맥락에서는 평가 및 검증의 지표로만 사용.
- **치팅** — 에이전트가 평가를 우회하는 행위.

### 병렬
- **작업 병렬** — 서로 다른 작업을 여러 서브에이전트가 나누어 수행. 금지.
- **관점 병렬** — 같은 산출물을 여러 모델이 교차검증하는 구조. 허용.

## 미정 항목 처리

각 섹션의 **구체화 예정** 과 **미정** 은 사용자 결정을 기다리는 열린 질문.

해당 경우 원칙(WHY)에 근거해 판단하고, 자의적 수치·템플릿·프로토콜을 만들어 끼우지 말 것.

## 참고: 미채택한 원칙

### Karpathy "Simplicity First" — 별도 보존

`forrestchang/andrej-karpathy-skills` (https://github.com/forrestchang/andrej-karpathy-skills) 의 네 원칙 중 **Simplicity First** 는 patchwork 에 직접 통합하지 않는다.

**사유**:
- patchwork 의 두 축(맥락 보존·독립 검증)과 직교. 간결성은 다른 축.
- "200줄 → 50줄이면 rewrite" 같은 수치 기준은 patchwork 의 "자의적 수치·템플릿 금지" (위 § 미정 항목 처리) 와 맞지 않음.

**운영 방법**: 필요 시 사용자가 CLAUDE.md 레벨에 별도로 기록. patchwork SKILL.md 에는 포함하지 않는다.

### Karpathy 원칙 4 중 "Loop independently" — 기각

같은 출처의 Goal-Driven Execution 에 포함된 "loop independently" 구절은 **자기 평가 허용**으로 해석 가능해 patchwork §5 (서로 다른 2 평가자) 와 정면 충돌하므로 채택 제외.

**참고로 통합된 부분**: Karpathy 원칙 1 (Think Before Coding) 은 SKILL.md §0 으로 신설, 원칙 3 (Surgical Changes) 은 §2 "변경 범위 외 불개입" 소절로 차용, 원칙 4 의 "요청 변환 템플릿" 은 §4 끝에 "요청 → 검증 가능 목표 변환" 소절로 차용.

**원칙 4 변환 템플릿의 재검토 기록**: 초기에는 §0·§4·§5 와 충돌로 판단해 보류했으나, "언제 TDD 를 적용할 것인가" (patchwork §4) 와 "적용된 요청을 어떻게 테스트 형태로 옮길 것인가" (Karpathy 변환 템플릿) 는 서로 다른 차원임이 드러나 재채택. §0 → §4 판정 → 변환 템플릿 → §5 전달 의 순차·조건부 관계로 정리됨.
