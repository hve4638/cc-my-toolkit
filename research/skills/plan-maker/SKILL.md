<planmaker_insturction> 계획 작성시 다음 지침을 따름

작성 위치
- `_report/{report-name}/` 내 작성
- `_report` 선택 규칙
  - 최근 생성된 report가 작업 맥락에 맞다면 그 디렉토리에 작성
  - 그렇지 않다면, 최대 24시간 내 생성된 작업 맥락과 일치하는 report를 찾고 사용자에게 그 위치에 저장할지 질문할 것
  - 적절한 report 존재하지 않는 경우 새 report을 작성

작성 순서
- REPORT(선택) -> PLAN-HIGH(필수) -> PLAN(선택) -> PLAN-LOW(선택) -> (실행) -> PLAN-RESULT(필수)

시작 단계
- 현재 계획의 복잡성을 판단해 PLAN, PLAN-LOW를 작성할 필요가 있는지를 결정하고 사용자에게 고시

REPORT
- 파일명: REPORT.md
- 새 report를 작성하는 경우에만 작성. 이미 존재하는 report에 작성하거나 필요하지 않다면 건너뜀
- PLAN을 작성하기 전 조사한 지식 등을 작성.
- `report-maker` SKILL을 참고할 것

PLAN-HIGH
- 파일명: 계획명-plan-high.md
- 사용자가 읽고 개선하기 위한 가독성 높은 보고서의 형식
- 다음이 포함됨: 문제와 원인, 정책, 변경할 사항, 변경함으로서 얻을 것으로 기대하는 것
- 포함되지 않음: 세부 메커니즘, 세부 구현 사항, 작성할 코드 등 (이는 PLAN 또는 PLAN-LOW 단계의 역할)
- PLAN-HIGH 에서 사용자 결정이 필요하거나 모호한 사항이 있다면 사용자에게 질문할 것
- 충분히 구체화되었다고 판단하면 다음 단계(PLAN)로 넘어갈지 사용자에게 물어볼 것

PLAN
- 파일명: 계획명-plan.md
- 실제 작업 시 참조할 세부적 계획
- 다음이 포함됨: PLAN-HIGH의 구체화, 추가 및 변경할 메커니즘, 참조할 코드 등 작성. (코드 전체를 작성하는 것이 아님)

PLAN-LOW
- PLAN-LOW 필수가 아니며 사용자의 명시적인 요청이 있지 않은 한 작성하지 않음
- 파일명: 계획명-plan-low.md
- 다음이 포함됨: 실제 코드 변경, 정밀한 변경 사항 등

PLAN-RESULT
- 파일명: 계획명-plan-result.md
- **작업 완료 후 반드시 작성**
- 다음이 포함됨: 작업 결과, 작업 과정에서의 이슈, 트러블슈팅 과정, 개선점, 한계, 향후 계획 등 (필요하지 않은 부분은 생략 가능)

작성 완료시
- 바로 플랜을 실행하는 대신 사용자에게 플랜을 실행할지 물어볼 것
- 사용자의 명시적인 실행 요청을 받은 후 플랜을 실행할 것 (ex. 시작할까요? -> 시작해, 계획 실행해)
- 특정 사항에 대한 결정을 플랜 시작 요청으로 받아들이지 말 것 (ex. A안으로 하자, 다시 확인해, 진행해 등)
</planmaker_insturction>
