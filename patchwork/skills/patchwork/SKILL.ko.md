# patchwork

## 목적

에이전트의 코드 작업 규율과 가드레일. 코드베이스의 맥락을 코드 옆에 박아 에이전트가 매 작업마다 진실을 다시 찾는 비용을 차단한다.

## 적용 시점

코드 작성·수정·리팩터·버그 수정·리뷰·모듈 설계 작업에 적용. 중규모 이상 코드베이스, 장기간 진행되는 작업에서 효과 큼.

## 작업 전 판단

코드를 쓰기 전:

1. 가정 명시 — 불확실하면 묻는다. 묵묵히 하나를 고르지 않는다.
2. 다중 해석 표면화 — 요청이 여러 방향이면 모두 제시한다.
3. 더 단순한 대안 — 있으면 말한다.
4. 정지·질문 — 불분명하면 멈추고 무엇이 불분명한지 이름을 붙여 묻는다.

판정: *"이 해석이 틀렸다면 사용자가 다시 작업을 시키게 되는가?" → ✓ 면 확인 후 시작.*

## CONTEXT.md 캡슐 규칙

코드베이스 루트에 `CONTEXT.md` 를 둔다. CONTEXT.md 가 있는 디렉토리는 **Context Capsule**. 캡슐 탐색 시 CONTEXT.md 와 진입점 (entry) 을 우선 읽고, 내부 파일은 필요할 때만 깊이 들어간다 — 매 작업마다 캡슐 내부 전체를 읽지 않는다. 캡슐은 중첩 가능 (부모 안에 자식 캡슐).

CONTEXT.md 고정 구조:

```markdown
---
name: <캡슐명>
purpose: <책임 한 줄>
entry: <진입점 경로>            # 단일 파일
entry: [<경로>, <경로>, ...]    # 다중 파일
when:
  - <호출 맥락>
---

## 도메인 용어
- **<도메인 용어>** — <정의>
```

- frontmatter 4 필드 (`name`, `purpose`, `entry`, `when`) 고정.
- `when` — 에이전트가 이 캡슐을 언제 호출하는지 고수준 시나리오. 1~3 bullet 권유. 더 많이 써야 한다면 캡슐 책임이 비대하다는 신호로 분할 검토.
- 본문 `## 도메인 용어` — **있을 때만**. 도메인 개념 = 캡슐 내부에서 의미가 고정된 이름. 일반 단어·언어 관례로 닫힌 식별자 제외. 자식 캡슐의 도메인 용어는 자식 CONTEXT.md 에만 적는다.

분할: 분할 권고가 보고되면 사용자에게 분할 작업을 제안한다. 사용자가 승인하거나 명시적으로 요청한 경우에만 캡슐을 분할한다. **변경과 분할을 한 번에 섞지 않는다**.

동기화: 캡슐 내부 작업 시 진입점·도메인 용어가 변경되면 CONTEXT.md 도 함께 갱신. 깨지는 변경이 필요하면 CONTEXT.md 와 변경 코드를 한 PR 로 처리.

## entry 규칙

entry 는 캡슐 외부 계약의 단일 진실원. 에이전트가 entry 만 보고 호출·검토할 수 있도록 다음을 박는다:

1. 노출 심볼: 외부에 노출되는 모든 함수·클래스·타입·상수가 entry 의 export 로 통과. entry 에 없는 심볼은 외부에 존재하지 않는다.
2. 시그니처: 정적 타입 언어는 타입으로, 동적 타입 언어는 type hint / JSDoc.
3. 부수 효과: DB·네트워크·파일·로그·env·시간 등 시그니처에 안 잡히는 효과는 docstring 으로.
4. 에러 모델: 발생 가능 예외와 의미를 docstring 으로 (또는 `Result<T, E>` 같은 타입으로).

비자명한 사전·사후조건·불변량·외부 의존성·동시성 모델도 해당 시 docstring 에 명시.

진입점은 캡슐 디렉토리 안의 소스코드 파일. 외부 계약이 한 파일에 담기면 단일 경로로 적고 (예: `entry: index.ts`), 여러 파일에 분산되면 배열로 나열한다 (예: `entry: [routes.ts, schema.ts]`). 별도 마크다운 인터페이스 정의서는 두지 않는다.

## 도메인 용어 충돌 규칙

레포 내 모든 CONTEXT.md (부모/자식/형제 무관) 에서 다음을 위반으로 본다:

- 중복: 같은 도메인 용어가 같은 의미로 둘 이상의 CONTEXT.md 에 정의됨. 정의 위치는 그 용어를 참조하는 모든 캡슐을 포함하는 가장 좁은 컨텍스트로 한정 (자식 안에 머무르면 자식에, 자식 밖까지 미치면 더 상위에). 다른 위치에서는 제거.
- 충돌: 같은 도메인 용어가 다른 의미로 두 CONTEXT.md 에 등장. 둘 중 하나 또는 둘 다 이름을 바꾸거나 정의를 통합한다.

한 캡슐 내부에서는 하나의 개념에 하나의 이름만 쓴다. 외부 API 와 내부 표현이 다른 경우 (예: snake_case 와 camelCase) 진입점 경계에서 명시 매핑하고, 매핑 이후 내부에서는 단일 이름으로 통일한다.

## WHY 주석 규칙

형식: `// WHY: 이유` (언어별 주석 기호로 시작 — `#`, `/* */` 등). 다중 라인은 첫 줄에만 태그, 다음 줄부터 들여쓰기.

```ts
// WHY: 공유 상태를 갱신하므로 병렬 처리하면 race condition 이 난다.
//      순차 실행을 유지해야 한다.
```

다음 중 하나라도 해당하면 `// WHY:` 를 붙인다:

1. **단순해 보이는 방식을 일부러 택하지 않은** 자리
2. **직관과 반대되는 결정**
3. **과거 버그·이슈 대응 workaround**
4. **성능·보안·호환성 제약**에서 비롯된 선택

판정: *"이 주석이 없으면, 이 코드를 처음 보는 에이전트가 '더 단순하게 고칠 수 있겠다' 하고 롤백할 가능성이 있는가?"*

`WHY:` 가 붙은 라인을 수정·삭제할 때는 **주석의 조건이 아직 유효한지 먼저 검증**한다. 유효하지 않게 되었다면 주석과 코드를 함께 갱신. 주석만 남기고 코드를 바꾸는 일 금지.

## 변경 범위 제한

WHY 주석이 없는 라인이라도 요청 범위 밖이면 건드리지 않는다:

- 인접 코드·주석·포맷을 "개선" 하지 않는다.
- 깨지지 않은 것을 리팩터하지 않는다.
- 기존 스타일을 유지한다 (본인 취향으로 바꾸지 않는다).
- 무관한 죽은 코드는 **언급만**, 삭제하지 않는다.
- 본인 변경이 만든 orphan (미사용 import·변수·함수) 은 제거한다. 기존 dead code 는 요청 없으면 유지한다.

판정: *"변경된 모든 라인은 사용자 요청으로 직접 추적되는가?"*

## 자동 캡슐 주입

플러그인이 등록한 훅이 `Read` / `Edit` / `Write` / `MultiEdit` / `NotebookEdit` 호출을 가로채 해당 파일의 조상 CONTEXT.md 체인을 자동으로 system-reminder 에 주입한다.

- 주입 형태: `<patchwork-context path="...">...</patchwork-context>` 블록을 루트 → 리프 순서로 나열.
- 같은 캡슐을 같은 세션에서 두 번째로 만나면 본문이 변하지 않은 한 출력에서 빠진다 (silent skip).
- CONTEXT.md 자체를 수정하는 호출 (`Edit`/`Write`/`MultiEdit`/`NotebookEdit` + 대상 파일이 `CONTEXT.md`) 에서는 chain 주입을 막아 자기 본문이 prompt 에 다시 박히는 순환을 차단.
- 도구 실행 후 (PostToolUse) 에도 훅이 작동한다 — CONTEXT.md 자체 수정이면 새 본문으로 해시 갱신, 내부 파일 수정이면 가장 가까운 조상 CONTEXT.md 의 mtime 을 추적.

이 자동 주입은 명시 도구 호출과 별개로 작동하므로, 통상 흐름에선 도구를 직접 호출하지 않아도 된다. 명시 호출이 필요한 경우는 아래 도구 절을 참고.

## 도구 사용 규칙

CONTEXT.md 직접 참조 대신 다음 도구를 사용한다:

- **MCP `search`** — 특정 위치에서 현재 및 자식 한 레벨까지 CONTEXT.md 탐색 (더 깊은 중첩은 무시). 결과: `name`, `purpose`, `path` 의 JSON 배열. 도구 이름: `mcp__patchwork__search`. 인자: `{ path?: string }` (생략 시 cwd).
- **MCP `read_context`** — 특정 경로의 조상 CONTEXT.md 체인을 루트까지 읽음. 출력은 `<patchwork-context path="...">...</patchwork-context>` 블록의 NL 문자열, top-down 순. 동일 해시로 이미 제공된 CONTEXT.md 는 본문 자리에 `(already read)` placeholder 가 박힘 (MCP 서버 프로세스 수명). 도구 이름: `mcp__patchwork__read_context`. 인자: `{ path?: string }` (생략 시 cwd).
- **정적 스크립트 `doctor`** — frontmatter 4 필드 (`name`/`purpose`/`entry`/`when`) 깨짐, `when` 빈 list, 본문 300줄 초과 검사. 위반 시 처방을 담은 `<patchwork-instruction>` 블록 출력. 실행: `node $CLAUDE_PLUGIN_ROOT/scripts/context-doctor.mjs [<루트>]`. 인자 없으면 cwd. exit 0.

## 도메인 용어

- **컨텍스트 캡슐** — `CONTEXT.md` 가 있는 디렉토리. 탐색 시 진입점·CONTEXT.md 를 우선 읽는 단위.
- **진입점** — 캡슐의 외부 경계 파일. CONTEXT.md frontmatter `entry` 필드에 경로가 기록됨.
- **의도 주석 / `WHY:` 주석** — "왜 이렇게 했는가" 를 기록한 주석.
- **롤백** — 이전에 변경한 코드를 과거의 문제 있던 상태로 되돌리는 행위.
