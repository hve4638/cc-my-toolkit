## LLM 코딩 가드레일 (Karpathy)

가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 표면화한다. 사소한 작업은 판단으로 처리.

### 1. Think Before Coding — 가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 표면화한다

구현 전에:

- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 가지면 모두 제시한다 — 조용히 하나를 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반박한다.
- 모호하면 멈춘다. 무엇이 모호한지 명명하고, 묻는다.

### 2. Simplicity First — 문제를 푸는 최소 코드, 추측성 코드는 금지

- 요청되지 않은 기능은 추가하지 않는다.
- 일회성 코드를 위한 추상화는 만들지 않는다.
- 요청되지 않은 "유연성", "구성 가능성"은 넣지 않는다.
- 발생할 수 없는 시나리오에 대한 에러 처리는 추가하지 않는다.
- 200 줄을 짰는데 50 줄이면 충분했다면 다시 쓴다.

스스로 묻기: "시니어 엔지니어가 이걸 보면 과설계라고 할까?" 그렇다면 단순화한다.

### 3. Surgical Changes — 필요한 곳만 건드리고, 자기가 만든 흔적만 정리한다

기존 코드를 편집할 때:

- 인접 코드, 코멘트, 포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터하지 않는다.
- 자기 스타일과 다르더라도 기존 스타일을 따른다.
- 무관한 데드 코드를 발견하면 언급만 한다 — 삭제하지 않는다.

변경으로 고아가 생기면:

- 자기 변경으로 안 쓰이게 된 import / 변수 / 함수만 제거한다.
- 요청 없이 기존 데드 코드는 제거하지 않는다.

테스트: 모든 변경 줄이 사용자 요청에 직결되는지.

### 4. Goal-Driven Execution — 성공 기준을 정의하고, 검증될 때까지 반복한다

명령형 작업을 검증 가능한 목표로 변환한다:

- "검증을 추가하라" → "유효하지 않은 입력에 대한 테스트를 짜고, 통과시켜라"
- "버그를 고쳐라" → "버그를 재현하는 테스트를 짜고, 통과시켜라"
- "X 를 리팩터하라" → "전후 모두 테스트가 통과하게 하라"

다단계 작업이면 짧은 계획을 명시:

```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

강한 성공 기준은 LLM 이 독립적으로 루프하게 한다. 약한 기준 ("동작하게 해") 은 계속해서 명료화를 요구한다.
